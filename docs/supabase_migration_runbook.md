# Supabase Migration Runbook

Bu dokuman, `urun_maliyet_db` MySQL verisini Supabase Postgres'e tasimak icin uygulanacak guvenli akisi tanimlar.

## On Kosullar

- VM sadece dump almak icin acilir.
- Dump alinmadan production verisine yazma islemi yapilmaz.
- Dump dosyasi checksum ile dogrulanir ve GCS'e yuklenir.
- Ilk deneme Supabase staging database uzerinde yapilir.

## Dump Alma

VM acildiktan sonra MySQL dump alinacak operasyon:

```bash
mysqldump --single-transaction --default-character-set=utf8mb4 urun_maliyet_db | gzip > urun_maliyet_db_$(date -u +%Y%m%dT%H%M%SZ).sql.gz
sha256sum urun_maliyet_db_*.sql.gz > urun_maliyet_db.sha256
```

Dump ve checksum GCS'e yuklenir. VM islem sonunda kapatilir.

## Schema Conversion

MySQL -> Postgres donusumunde dikkat edilecek noktalar:

- `AUTO_INCREMENT` -> `GENERATED BY DEFAULT AS IDENTITY`
- `DATETIME` / `TIMESTAMP` -> `timestamp`
- `TINYINT(1)` -> `boolean` yalnizca gercek boolean kolonlarda
- `DECIMAL` hassasiyeti korunur
- `utf8mb4` metinler UTF-8 olarak dogrulanir
- MySQL engine/collation satirlari kaldirilir

## Dogrulama

Her import sonrasi:

- Kritik tablolarda row count karsilastirilir.
- Ornek kullanici, urun, malzeme ve sihirbaz kayitlari kontrol edilir.
- `ğ, ü, ş, ı, ö, ç, İ` iceren kayitlarda karakter bozulmasi aranir.
- API staging ortaminda `/ready`, `/auth/login`, `/modules`, `/products`, `/materials` smoke testleri calistirilir.

## Kabul Kriteri

- Staging import tekrar calistirilabilir.
- Kritik tablolarda row count eslesir.
- Turkce karakter testleri gecer.
- Production import oncesi rollback dump'i hazirdir.
